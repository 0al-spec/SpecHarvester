# P27-T5 — Real Repository Author-Ready Draft Calibration Matrix

## Objective

Run and document an author-ready calibration matrix over real repository
checkouts so P27 can answer a product question, not only a plumbing question:

```text
How much author work remains after SpecHarvester produces a valid starter
package?
```

P27-T1 through P27-T4 define the quality bar, machine verdict, stop policy, and
author review surface. P27-T5 should calibrate that surface across several real
repository shapes and record when remaining work is normal author curation
versus repeated generator weakness.

## Scope

In scope:

- Define a machine-readable author-ready calibration matrix artifact.
- Derive matrix rows from existing `SpecHarvesterRealRepositoryQualityReport`
  output plus optional author review notes.
- Track repository/package identity, ecosystem shape, author-ready status,
  estimated author edit count, edit categories, evidence gaps, weak claims, and
  recommended generator follow-ups.
- Document how to run the matrix over local real repository checkouts without
  committing `.smoke` outputs.
- Record a local observed calibration run over available real checkouts.
- Update GitHub docs, DocC, Workplan, and docs-contract tests.

Out of scope:

- Changing generated `specpm.yaml` or `specs/*.spec.yaml`.
- Publishing accepted SpecPM packages.
- Running harvested package scripts, builds, tests, package managers, or
  dependency installers.
- Requiring LM Studio or any LLM provider.
- Treating matrix scores as registry acceptance or upstream endorsement.

## Test-First Plan

| Test | Purpose | Expected Result |
| --- | --- | --- |
| Calibration builder | Verify matrix derivation from quality report and notes. | Matrix includes package rows, summary counts, edit totals, categories, and follow-up recommendations. |
| CLI output | Verify operator command. | `author-ready-calibration-matrix` writes stable JSON from local quality report input. |
| Docs contract | Verify public docs. | GitHub docs, DocC, roadmap, Workplan, and `next.md` mention the P27-T5 matrix and boundaries. |
| Real smoke | Verify local run path. | A local `.smoke` calibration run can produce a matrix from real repository artifacts without committing outputs. |

## Implementation Plan

1. Inspect current real-repository validation and quality-report contracts.
2. Add a small calibration matrix builder that reads quality-report JSON and
   optional author-review notes JSON.
3. Expose it through the CLI as `author-ready-calibration-matrix`.
4. Add docs and DocC pages describing matrix schema, command usage, local run
   pattern, scoring, and boundary rules.
5. Run the existing real-repository pipeline on available local checkouts where
   practical and record an observed P27-T5 matrix summary.
6. Run quality gates and archive the Flow task.

## Acceptance Criteria

- A versioned `SpecHarvesterAuthorReadyCalibrationMatrix` JSON artifact is
  documented and generated by tests.
- The matrix records author edit counts and categories separately from SpecPM
  validation status.
- Repeated gaps can be routed to generator follow-up work without implying that
  every author edit should become automation.
- Docs state that generated `.smoke` outputs remain uncommitted local evidence.
- Safety boundaries remain explicit: no repository code execution, no package
  manager execution, no registry publication, and no acceptance authority.
- P27-T5 is archived with validation results and a review artifact.
